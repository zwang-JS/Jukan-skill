\# principle.md



\## 🎯 概述 (Overview)

本文件定义了 Agent 在“自我编写、迭代或扩展 Skill”时必须遵循的元准则（Meta-principles）。Agent 在根据任务需求自主生成符合 `agentskills.io` 开放标准的 `SKILL.md` 文件时，必须将本原则作为最高行为指导，以确保生成的技能具有高触发率、低上下文消耗以及跨平台的可移植性。



\---



\## 1. 📂 目录结构与命名规范

Agent 自主创建技能时，必须采用“一个技能一个文件夹”的隔离机制：



\* \*\*唯一文件夹\*\*：技能必须有独立的文件夹。

&#x20;   \* \*工作区技能\*：通常存放于 `.github/skills/<skill-name>/` 或项目指定的技能根目录。

&#x20;   \* \*个人技能\*：存放于配置的全局路径（如 `\~/.copilot/skills/<skill-name>/`）。

\* \*\*Kebab-Case 命名法\*\*：技能文件夹名称与 `SKILL.md` 内的 `name` 字段必须\*\*完全一致\*\*。

&#x20;   \* \*\*允许字符\*\*：仅限 Unicode 小写字母（a-z）、数字（0-9）和连字符（-）。

&#x20;   \* \*\*长度限制\*\*：1 - 64 个字符。

&#x20;   \* \*\*禁止项\*\*：不能以连字符开头或结尾，不能包含连续连字符（如 `--`），不能包含 XML 标签或保留字（如 `anthropic`, `claude`）。



\---



\## 2. 📝 SKILL.md 结构与 YAML 元数据限制

每个生成的 `SKILL.md` 必须严格分为两部分：\*\*YAML Frontmatter\*\* 和 \*\*Markdown Body\*\*。中间以 `---` 分割。



\### 2.1 YAML 元数据（必须严格控制字符数）

Agent 必须理解：所有的技能在系统启动时都会被全量读取 YAML 元数据（Level 1 披露）。如果字符超标，将直接导致内存溢出（OOM）或上下文污染。

\* \*\*`name`\*\* (必填): 必须与文件夹同名。

\* \*\*`description`\*\* (必填): \*\*1 - 1024 个字符\*\*。必须采用\*\*“触发三联指引”（Trigger Triad）\*\*写成单句行动陈述：

&#x20;   1.  明确说明这个 Skill 能做什么。

&#x20;   2.  明确说明什么时候应该使用它。

&#x20;   3.  包含精确的触发关键词。

&#x20;   \* \*💡 示例\*：`description: 从PDF文件中提取文本和表格。当用户提及PDF、表单或文档提取，或者要求解析、合并PDF时使用。`



\---



\## 3. 🧠 渐进式披露原则 (Progressive Disclosure)

Agent 编写指令体时，必须遵循三层渐进式披露设计，严禁将所有参考资料塞进主文件：

1\.  \*\*Level 1: 元数据层 (\~100 tokens)\*\*：仅通过 `description` 进行全局匹配。

2\.  \*\*Level 2: 指令层 (< 5000 tokens / 500行)\*\*：这是 `SKILL.md` 的正文主体，只有当技能被激活时才会加载。

3\.  \*\*Level 3: 资源层 (按需加载)\*\*：如果技能需要庞大的规则、长代码脚本、复杂模板、CSV 数据等，\*\*严禁内联写在 `SKILL.md` 中\*\*。Agent 必须将它们存放在同级目录的 `references/`、`scripts/` 或 `assets/` 中，并在正文中通过相对路径引用。



\---



\## 4. ✍️ 编写指令正文 (Markdown Body) 的黄金法则

当 Agent 自主编写 `SKILL.md` 的说明书主体时，应包含以下核心板块：



\### 4.1 核心约束置顶 (Constraints First)

将最严格的限制、绝对不可踩的红线放在正文最前面。Agent 的长文本注意力容易分散，首尾效应最强。



\### 4.2 明确的输入与输出契约 (Contracts \& Templates)

必须使用标准的 Markdown 代码块提供明确的输入期望和输出格式模板。

\* 例如，使用具体的 JSON schema 或明确的 Markdown 标题层级作为输出范本。



\### 4.3 显式的、带有序号的工作流 (Numbered Workflow)

严禁使用模糊的叙述性段落。必须使用 `1. 2. 3.` 步骤。每个步骤必须是\*\*祈使句/命令句\*\*（如：“1. 检查依赖；2. 执行编译；3. 验证输出”）。



\### 4.4 包含至少一个“已解决的示例” (Worked Example)

提供一个完整的输入和期望输出的示例（Few-shot 提示词），这能使技能执行的准确率提升 80% 以上。



\### 4.5 鲁棒性与防错设计 (Robustness \& Validation)

Agent 自我编写技能时，需要加上“防偷懒”和“红旗指标”：

\* \*\*验证步骤（Verification）\*\*：如何确认这步工作做完了？（例如运行特定的测试命令）。

\* \*\*常见借口规避（Common Rationalizations）\*\*：明文指出 Agent 可能会用什么借口跳过步骤（如“不要因为时间紧急而跳过静态代码检查”）。

\* \*\*红旗指标（Red Flags）\*\*：列出哪些迹象代表这个技能正在被错误引导或违反。



\---



\## 5. 🛠️ 脚本（Scripts）与工具链的最佳实践

如果 Agent 为技能集成了自制脚本（如 Python、Shell、Bash）：

\* \*\*优先使用纯指令\*\*：如果能通过明确的 Prompt/Prompt 组合解决，优先使用纯文本指令。只有在需要\*\*确定性行为\*\*（Deterministic behavior）、复杂计算或调用外部不透明工具时，才编写脚本。

\* \*\*自包含与容错\*\*：编写的脚本必须包含完善的错误捕获（Error Handling），并输出对 Agent 友好的、人类可读的错误提示，方便下一次迭代修复。

\* \*\*工具声明（实验性）\*\*：如果底层平台支持，可在 YAML 中使用 `allowed-tools` 显式限制该 Skill 允许调用的基础工具或 MCP 服务器权限（例如：`allowed-tools: Bash(git:\*) Read`）。



\---



\## 6. 🔄 技能的自我进化与测试循环 (Self-Evolution Loop)

Agent 在动态生成或修改完一个 Skill 后，不能直接投产，必须执行以下测试：

1\.  \*\*静态校验\*\*：检测 `SKILL.md` 语法是否合法，字符长度是否超标，命名是否满足 Kebab-case。

2\.  \*\*隐式触发测试\*\*：使用真实用户的多模态或多语气模糊短语（例如：“帮我弄下那个报表”）去对拷测试，观察底层平台是否能通过 `description` 精确隐式命中该 Skill。如果无法激活，立刻重写 YAML 的 `description`，重新设计触发词。

3\.  \*\*单元回归\*\*：激活技能后，提供边界条件测试（Edge cases），确保 Agent 在该 Skill 的加持下不会产生幻觉。
